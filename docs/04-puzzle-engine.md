# Puzzle engine: generation, solver, difficulty

## Puzzle structure

- Production target: 5 suspects × 4 categories (flair, time, object, location) with 5 values each.
- Reference scale of the puzzle (engine [src/server/engine.ts](../src/server/engine.ts)): reduced **4×3** (flair, time, object) - same logic as in production.
- The player links suspect ↔ value of each category. The **time** category is ordered - the before/after clues operate on it.

## Core principle: logic is code, flavor is a separate layer

Logic and difficulty are generated by **deterministic code**. Flavor (theme, names, characters, objects, backstory) comes as a separate layer - from a curated set of templates (optionally with AI). **AI does not touch the logic/difficulty itself.**

Why we do NOT hand logic to AI:
- whatever the AI produces, it still has to be run through the solver (uniqueness + difficulty) - and it cannot reliably hit both targets, which means almost all of its output is rejected and regenerated; `seeded shuffle` + solver does the same thing more cheaply, more deterministically, and more reliably;
- AI does not guarantee a unique solution;
- AI does not "feel" the difficulty profile - only the solver measures it;
- seed reproducibility (`subreddit:date:theme`) requires determinism.

Key shift: **difficulty/branching is not "content" that gets written, but an emergent property of the set of clues.** You select clues and measure with the solver. So generating difficulty is a search, not a creative act.

## Pipeline (all code, except step 6)

1. Take theme/characters/objects from the curated set; anything missing - safe defaults.
2. Hidden solution: seeded shuffle of each category.
3. Pool of all clues that are true given the solution (templates below).
4. Select a subset for the **target tier**: brute-force for uniqueness + solver for tier.
5. Render clues into text (templates, [src/shared/render.ts](../src/shared/render.ts)).
6. Apply the flavor (names/theme/backstory from a template).

Seed: `subredditId:date:winningThemeId` - the puzzle is deterministic and reproducible. We still store the solution (protection against desync if the generator changes).

## Solver: two player models

Uniqueness - brute-force/backtracking (size is small: 4×3 → 24³; 5×4 → fast with pruning; cut off after the 2nd solution).

"Whether it is solvable without guessing" depends on which techniques are available to the player. Two models:

- **WEAK** - domains per suspect (only suspect×category grids). Exactly what can be marked on the current board.
- **GRID** - all pairwise value×value grids (flair×time, etc.) + transitivity. This is how a meticulous logic-grid player reasons.

"forced under a model" = there exists a path to the solution without a single guess using these techniques.

## Difficulty = TIER (the spread of the two models)

| Tier | Condition | Meaning |
|---|---|---|
| 🟢 green  | weak-forced | solvable on the simple board without guessing |
| 🟡 yellow | grid-forced, but NOT weak | needs cross-referencing logic; NO guess required |
| 🔴 red    | unique, but not grid-forced | a guess is genuinely required (search / "by contradiction") |

**Default for daily - 🟡:** there is always a clean logical path (fair, no guessing), but you have to link flair↔time↔object rather than reading across suspect rows. 🟢 - a warmup, 🔴 - hardcore for the smart ones.

> The old benchmark "difficulty = number of clues" (Easy 6-8 / Medium 8-10 / Hard 10-12) is **obsolete**: the number of clues does not determine difficulty. The pair of models determines the tier.

### Caveat about the board

🟡 requires cross-referencing (value×value) deductions. On the current board (only suspect×category) there is nowhere to mark them - the player keeps them in their head. A motivated player solves it without guessing (verified on 20 independent runs), an impatient one guesses even though guessing is not required. To make 🟡 more legible - you could also show the value×value grids on the board (a full logic-grid UI). This is a UX choice, it does not change the logic.

### A dead end we already checked (do not re-investigate)

A smooth "medium" model between WEAK and GRID via "transitivity depth" **does not work out**: the boundary is effectively binary (you either run value×value grids or you don't), a depth meter does not separate the cases and does not even respect "easy = shallow depth". We use the binary spread of the two models.

## Clue templates

| Type | Form | Example |
|---|---|---|
| not_equal | `("ne", s, cat, v)` | John did not post at 15:00 |
| same_owner | `("same", c1,v1, c2,v2)` | the owner of the green flair posted a sock |
| not_same_owner | `("nsame", c1,v1, c2,v2)` | the pizza was not posted by the owner of the blue one |
| before | `("before", refA, refB)` | the owner of the red one posted earlier than John |

`ref = ("s", name)` (suspect) or `(cat, value)` (value). The metaphor is uniform - **"posted"** (an instantaneous event), not "was online" (an interval): both `before` and `"not at T"` read unambiguously on first reading (see [decision log](09-design-decisions.md), #11).

## Engine implementation

Production engine in TypeScript (Devvit); the original Python reference was removed, the models ported one-to-one:
- [`src/server/engine.ts`](../src/server/engine.ts) - `Puzzle` (WEAK + brute-force), `Grid` (GRID), `classify()`, `generate(tier)`.
- [`src/shared/render.ts`](../src/shared/render.ts) - clues → text.
- [`src/server/engine.test.ts`](../src/server/engine.test.ts) - parity test: generation of each tier + anchors Case #0 (🟡) and Case #1 (🟢).

## Simplification for the hackathon

~10 template puzzles in case the generator fails; the curated set of templates supplies names/items/theme; the solver checks uniqueness + tier; fallback to a prepared puzzle. More reliable than a perfect generator from scratch.
