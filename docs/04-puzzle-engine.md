# Puzzle engine: генерация, solver, сложность

## Структура задачи

- Продакшен-цель: 5 suspects × 4 категории (flair, time, object, location) по 5 значений.
- Reference-масштаб задачи (движок [src/server/engine.ts](../src/server/engine.ts)): уменьшенный **4×3** (flair, time, object) - та же логика, что в продакшене.
- Игрок связывает suspect ↔ значение каждой категории. Категория **time** упорядочена - на ней работают улики before/after.

## Главный принцип: логика - код, флейвор - отдельный слой

Логику и сложность генерирует **детерминированный код**. Флейвор (тема, имена, персонажи, предметы, легенда) приходит отдельным слоем - из курируемого набора заготовок (опционально с ИИ). **ИИ в саму логику/сложность не лезет.**

Почему логику НЕ отдаём ИИ:
- что бы ни выдал ИИ, его всё равно надо прогнать через solver (уникальность + сложность) - а попасть в оба таргета надёжно он не может, значит почти весь вывод бракуется и перегенерируется; `seeded shuffle` + solver делает то же дешевле, детерминированнее и надёжнее;
- ИИ не гарантирует единственность решения;
- ИИ не «чувствует» профиль сложности - его измеряет только solver;
- seed-воспроизводимость (`subreddit:date:theme`) требует детерминизма.

Ключевой сдвиг: **сложность/ветвление - не «контент», который пишут, а эмергентное свойство набора улик.** Отбираешь улики и измеряешь solver'ом. Значит генерация сложности - это поиск, а не творчество.

## Пайплайн (всё код, кроме шага 6)

1. Взять тему/персонажей/предметы из курируемого набора; недостающее - safe defaults.
2. Скрытое решение: seeded shuffle каждой категории.
3. Пул всех истинных при решении улик (шаблоны ниже).
4. Отбор подмножества под **целевой тир**: brute-force на уникальность + solver на тир.
5. Рендер улик в текст (шаблоны, [src/shared/render.ts](../src/shared/render.ts)).
6. Надеть флейвор (имена/тема/легенда из заготовки).

Seed: `subredditId:date:winningThemeId` - puzzle детерминированный и воспроизводимый. Solution всё равно сохраняем (защита от рассинхрона при изменении генератора).

## Solver: две модели игрока

Уникальность - brute-force/backtracking (размер мал: 4×3 → 24³; 5×4 → с pruning быстро; обрыв после 2-го решения).

«Решается ли без догадок» зависит от того, какие техники доступны игроку. Две модели:

- **WEAK** - домены на подозреваемого (только сетки suspect×категория). Ровно то, что можно отметить на текущем борде.
- **GRID** - все попарные сетки value×value (flair×time и т.п.) + транзитивность. Так рассуждает дотошный logic-grid игрок.

«forced под моделью» = существует путь к решению без единой догадки этими техниками.

## Сложность = ТИР (вилка двух моделей)

| Тир | Условие | Смысл |
|---|---|---|
| 🟢 green  | weak-forced | решается на простом борде без догадок |
| 🟡 yellow | grid-forced, но НЕ weak | нужна перекрёстная логика; догадка НЕ нужна |
| 🔴 red    | уникально, но не grid-forced | реально нужна догадка (перебор / «от противного») |

**Дефолт для daily - 🟡:** всегда есть чистый логический путь (честно, без гадания), но надо связывать флаир↔время↔предмет, а не читать по строкам подозреваемых. 🟢 - разминка, 🔴 - хардкор для умных.

> Старый ориентир «сложность = число улик» (Easy 6-8 / Medium 8-10 / Hard 10-12) **устарел**: число улик не определяет сложность. Тир определяет пара моделей.

### Оговорка про борд

🟡 требует перекрёстных (value×value) выводов. На текущем борде (только suspect×категория) их негде отметить - игрок держит в голове. Мотивированный решает без догадок (проверено на 20 независимых прогонах), нетерпеливый гадает, хотя гадать не обязательно. Чтобы сделать 🟡 нагляднее - можно показать на борде и сетки value×value (полный logic-grid UI). Это UX-выбор, логику не меняет.

### Тупик, который уже проверили (не искать заново)

Плавной «средней» модели между WEAK и GRID через «глубину транзитивности» **не получается**: граница по факту бинарная (ведёшь сетки value×value или нет), измеритель глубины не разделяет кейсы и не уважает даже «лёгкое = малой глубины». Используем бинарную вилку двух моделей.

## Шаблоны улик

| Тип | Форма | Пример |
|---|---|---|
| not_equal | `("ne", s, cat, v)` | John постил не в 15:00 |
| same_owner | `("same", c1,v1, c2,v2)` | владелец зелёного флаира постил носок |
| not_same_owner | `("nsame", c1,v1, c2,v2)` | пиццу постил не владелец синего |
| before | `("before", refA, refB)` | владелец красного постил раньше John |

`ref = ("s", name)` (подозреваемый) либо `(cat, value)` (значение). Метафора единая - **«постил»** (мгновенное событие), не «был онлайн» (интервал): и `before`, и `«не в T»` читаются однозначно с первого прочтения (см. [журнал решений](09-design-decisions.md), #11).

## Реализация движка

Продакшен-движок на TypeScript (Devvit); исходный Python-референс удалён, модели перенесены один-в-один:
- [`src/server/engine.ts`](../src/server/engine.ts) - `Puzzle` (WEAK + brute-force), `Grid` (GRID), `classify()`, `generate(tier)`.
- [`src/shared/render.ts`](../src/shared/render.ts) - улики → текст.
- [`src/server/engine.test.ts`](../src/server/engine.test.ts) - паритет-тест: генерация каждого тира + якоря Case #0 (🟡) и Case #1 (🟢).

## Упрощение для хакатона

~10 шаблонных пазлов на случай сбоя генератора; курируемый набор заготовок подставляет names/items/theme; solver проверяет uniqueness + тир; fallback на prepared puzzle. Надёжнее, чем идеальный генератор с нуля.
